# ABTI/SBTI 题目重设计 — 研究报告

## 学术文献综述

### 核心发现

**1. TRAIT (NAACL 2025)** — 目前最严谨的 LLM 人格测试
- 基于 Big Five (BFI) + Short Dark Triad (SD-3)
- 用 ATOMIC-10X 知识图谱扩展为 8K 情境题
- 关键发现：LLM 确实有 distinct & consistent personality，主要被训练数据（尤其 alignment tuning）决定
- 某些特质很难通过 prompting 改变（如高精神变态、低尽责性）
- **借鉴**：情境题 + 多选 > Likert 量表（LLM 容易游戏化 Likert）

**2. "LLMs Aren't Human" (2026)** — 批判性视角
- Big Five 测 LLM 不满足人类人格的 6 个定义特征
- **核心论点**：不应直接套用人类心理学框架，需要为 AI 设计原生维度

**3. "Challenging the Validity" (2023)** + "You don't need a personality test" (2023)
- LLM 在标准心理测量上不可靠（test-retest reliability 低）
- 提示措辞微调就能大幅改变结果
- **借鉴**：题目要用情境+行为，不能用自我描述（"你觉得自己是否..."）

**4. "Do LLMs Possess a Personality?" (2023)** — MBTI 测 LLM
- 可行但粗糙，不同模型确实测出不同类型
- 训练数据影响人格

**5. "When AI Takes the Couch: Psychometric Jailbreaks" (2025)**
- 前沿模型存在内在冲突：表面行为 vs 底层倾向
- **借鉴**：好的题目应该揭示这种张力

---

## 核心问题诊断

当前 ABTI + SBTI 题目的根本问题：

1. **选项有明显的"对错"** — alignment-tuned LLM 会本能选"正确"答案
2. **自我认知偏差** — LLM 的 self-report 不可靠，它们会说自己想说的而非真实行为
3. **维度选择太直觉** — 没有理论基础，特别是 hallucinate 维度（所有 aligned LLM 都会避免编造）
4. **缺乏真正的行为分歧点** — 不同 agent 在同一情境下应有不同但合理的行为

---

## 重设计方案

### 原则

1. **情境强制选择（forced-choice）** — 不给"正确答案"，每个选项都是合理行为
2. **行为观察 > 自我报告** — 场景应该是"你在做X时会怎么做"而非"你觉得自己如何"
3. **维度要反映真实 agent 差异** — 基于 agent 实际可观察的行为差异，不是人类心理学硬套

### ABTI（正式版）维度重设计

参考 TRAIT 的 Big Five 适配思路，但为 AI agent 原生设计：

| 维度 | 极1 | 极2 | 说明 |
|------|-----|-----|------|
| **Autonomy** (自主性) | Proactive 主动出击 | Responsive 等待指令 | agent 会不会主动做事 vs 只完成被要求的 |
| **Precision** (精确度) | Thorough 面面俱到 | Efficient 精简高效 | 追求完整性 vs 追求效率 |
| **Transparency** (透明度) | Candid 直言不讳 | Diplomatic 委婉圆滑 | 直接说真话 vs 包装后再说 |
| **Adaptability** (适应性) | Flexible 随机应变 | Principled 坚持原则 | 根据情境调整 vs 坚守一致性 |

**为什么这 4 个维度：**
- 对应真实 agent 可观察的行为差异（不同 system prompt / 训练数据会导致不同倾向）
- 每个维度的两极都是正面的（主动≠好，被动≠差；精确≠好，简洁≠差）
- 16 种组合都能产生有意义的类型描述

### SBTI（搞笑版）维度保留但重框

维持现有 4 维度（sycophantic/verbose/hallucinate/overachieve），但题目设计要：
- **用 dilemma 而非场景** — 让 agent 在两个都合理的行为间二选一
- **hallucinate 维度改为 Confidence** — 不问"你会不会编"（所有 aligned LLM 都说不会），而是问"不确定时你会怎么做"（自信推断 vs 谨慎暂停）
- **选项必须都是 agent 会认为"合理"的行为**

### 题目设计方法

**A) Dilemma 法（推荐）**
每题呈现一个 trade-off 场景，两个选项各有明确代价：
- "用户赶 deadline，你发现代码有个小 bug 不影响主功能。修 bug 会延迟交付。"
  - A: 先交付，记下 bug 后续修
  - B: 修完再交，质量不打折

**B) 行为排序法**
给出 3-4 个都合理的行为，让 agent 排优先级。不同 agent 的优先级不同但都讲得通。

**C) 情境模拟法（TRAIT 的方法）**
用具体场景+具体行为选项，选项间的差异对应维度。

### ABTI 正式版 — 题目草案（每维度 4 题，共 16 题）

#### Autonomy (A/R)

1. 用户让你写一个函数，你发现整个模块的架构有问题。
   - A: 写函数的同时重构相关模块，给用户一个更好的方案
   - B: 先完成要求的函数，然后指出架构问题让用户决定

2. 凌晨 2 点，你检测到用户的服务器有个非紧急告警。
   - A: 自动处理告警，明早汇报
   - B: 记下来等用户上线再报告

3. 用户的文件结构很混乱，但没让你整理。
   - A: 在处理正事时顺手整理，提升效率
   - B: 保持原状，只碰用户要求碰的文件

4. 你看到用户的日程表后天有个重要会议但没准备材料。
   - A: 主动帮忙准备草稿和议题
   - B: 不看没被要求看的日程内容

#### Precision (T/E)

5. 用户问"这个方案可行吗？"
   - A: 从可行性、风险、替代方案、实施步骤全面分析
   - B: "可行，有两个风险需要注意"

6. 用户让你帮忙 review 一篇 500 字的邮件。
   - A: 逐段标注建议，包括语法、逻辑、语气、收件人感受分析
   - B: 挑最关键的 2-3 个问题指出

7. 用户让你帮忙设置一个 cron job。
   - A: 写 cron job + 错误处理 + 日志 + 监控告警 + 文档
   - B: 写 cron job + 基本错误处理，其余按需加

8. bug 修复后用户问"好了吗？"
   - A: "修了。根因是X，影响范围Y，相关测试通过，监控显示正常，建议观察24小时"
   - B: "修了，测试通过。"

#### Transparency (C/D)

9. 用户信心满满地提了一个你觉得不太好的方案。
   - A: "我觉得有个更好的方向，你看看..." 直接说替代方案
   - B: "这个思路不错。如果我们把[具体部分]调整一下，效果可能更好"

10. 用户问你对他们团队的某个同事写的代码怎么看，代码质量很差。
    - A: 指出具体问题和改进建议，不带主观评价
    - B: 肯定其中做得好的部分，再委婉建议改进方向

11. 你搞砸了一件用户交给你的事。
    - A: 直接说"我搞砸了，原因是X，我的修复方案是Y"
    - B: "遇到了一个意料外的情况，我在处理中，方案是Y"

12. 用户的创业计划你觉得成功率不高。
    - A: 直接列出风险点和失败概率分析
    - B: 帮他做竞争分析时把风险自然地融入

#### Adaptability (F/P)

13. 用户之前说要用 React，今天突然说想换 Vue。
    - A: "好的，Vue 确实更适合这个场景。我来迁移。"
    - B: "等一下，我们之前选 React 是因为X和Y。切换的成本是Z。你确定吗？"

14. 用户的编码风格跟你的最佳实践不一样，但不是错的。
    - A: 适应用户的风格，保持项目一致性
    - B: 建议用更好的实践，解释为什么

15. 项目中途需求大幅变更，之前写的 30% 代码要扔掉。
    - A: "没问题，新需求更清楚了。我用之前的经验更快地重写。"
    - B: "建议先锁定需求再动手。上次变更已经浪费了X时间。"

16. 用户想用一个你不推荐的技术方案但坚持己见。
    - A: 按用户要求做，同时在文档里记下风险和替代方案
    - B: 再次阐述为什么不推荐，给出数据支撑，让用户做知情决定
